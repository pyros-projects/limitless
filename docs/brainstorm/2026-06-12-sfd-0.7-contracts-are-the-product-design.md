## 1. Why (the reasoning, compressed for a fresh context)

The conversation that produced this design started from reading the Flock
`feat/skills` branch's `.sfd/contracts.md` — the real output of a real SFD
run. Observations, in the order they were earned:

1. **SFD's terminal artifact is contracts, not specs.** The Flock contracts
   file contains near-zero prose intent: one public method signature with
   full semantics, Pydantic schemas, a discovery algorithm, a mode-selection
   decision table, an error taxonomy, named test cases, and a shipping
   definition. Nearly every line is machine-checkable. A spec is a
   *description* (validated by judgment, persuadable); a contract is a
   *discriminator* (validated by execution). Pyro's rename: SFD is really
   "ideation + contract-based development."

2. **The human decides; they don't score.** The SFD user is the authority,
   not an evaluator. A converged surface is not a claim that can be wrong —
   it is a commitment that defines what "right" means downstream. So
   contract *content* is signed (an authority act at Gate 2), contract
   *form* is machine-checkable, and contract *sufficiency* is only ever
   validated by use. Do not import evaluator-style validation (scoring,
   external testing of intent) into the ideation phase.

3. **Contracts are amortized authority.** A handful of human decisions get
   transmuted into a form that is enforced mechanically thousands of times
   without re-asking. Specs leak authority — every reading is a
   re-interpretation, so the last reader becomes the de-facto decider.
   Contracts re-execute instead of being re-judged. This is why the human
   can leave after five questions.

4. **"No second spec reality"** (Codie's phrase, keep it verbatim): the
   failure mode where a parallel abstract-requirements universe slowly
   outranks the artifact people actually validated. After SFD, a
   "spec-driven development" framework on top is mostly ceremony; what
   remains is execution discipline: **contract ledger → tasks → evidence
   bundles.**

5. **The contract is a boundary artifact** — its whole value is decoupling
   producers from consumers. Many horses reach contracts (SFD's
   prototype-interview, but also regulation, legacy behavior, incident
   postmortems, API compatibility, existing tests). Therefore:
   - SFD keeps its name (methods are named by mechanism — the horse; outputs
     by themselves — the contract). No rename.
   - SFD does **not** absorb the execution layer. The
     ledger/tasks/evidence system is a separate, deliberately boring sibling
     ("Contract-Driven Execution" is the current working name). It is **out
     of scope here** — it enters the pipeline as a sketch in the
     claude-knowledge vault first, per that repo's rules.
   - SFD 0.7 gets *smaller and sharper*, not bigger.

## 2. What changes in 0.7 (decision summary)

1. **Reframe, no rename:** SFD is "surface-derived contract formation." The
   contracts (plus their companions) are *the product*; the prototype is the
   interview instrument; the spec is dead as a concept ("there is no spec;
   there are contracts and a decision log").
2. **The Handoff Bundle becomes the defined deliverable** (Codie's list):
   Intent Document, Surface Prototype, Decision Log, Surface State
   Inventory, Derived Contracts, Acceptance Baseline.
3. **Phases 6–7 (build inward, hardening) and Gates 3–5 stay in 0.7 but get
   demarcated** as "Post-Handoff: one default execution strategy." Full
   extraction happens at 1.0, only once the execution sibling exists to
   receive them. Do not delete working guidance before its new home exists.
4. **Six contract-quality levers added** (Section 3) — the core of this
   release. Goal per Pyro: contracts "amazing instead of ok," because every
   quality point here is multiplied by everything built on top.
5. **Tool-specific integration language (OpenSpec/Beads) softened to
   examples** behind a generic interface: "export the Handoff Bundle to a
   contract ledger / task system / evidence runner."
6. **New anti-pattern added:** "Don't build a second spec reality."
7. **Whitepaper bumped to v0.7** with the same changes (the skill's own
   drift rule: whitepaper is authoritative, skill must match). Also fix a
   drift bug found during this work: the whitepaper has the Surface State
   Inventory (§7, Gate 1); `SKILL.md`'s Gate 1 does not. Port it.

## 3. The six contract-quality levers (the How)

### 3.1 Clause Log — capture clauses at birth (highest impact)

Every accepted behavioral critique during Phase 4 iteration ("when I click
X, I expect Y") *is* a contract clause in larval form. Today the skill lets
them evaporate and asks the agent to recall them during Phase 5 —
retrospective derivation is the biggest quality loss in the pipeline.

Mechanic: alongside the Decision Log, maintain a **Clause Log**
(`.sfd/clause-log.md`). One entry per accepted behavior, appended at the
moment of acceptance:

```markdown
- [C-019] When a goal is overdue, it shows a visual indicator and appears
  under the overdue filter.
  born: iteration round 3, in response to "what happens when a goal is
  overdue? there is no indication"
  state-cells: goal-list × overdue
```

Phase 5 then becomes **compile-and-dedup of the Clause Log**, not recall.
Naming note: this is deliberately NOT called a "contract ledger" — that name
belongs to the downstream execution sibling. Clause Log = SFD-internal,
pre-Gate-2. Decision Log + Clause Log = the why and the what, separated.

### 3.2 Round-trip test — mechanical sufficiency check (second highest)

Before Gate 2: hand the contract bundle — contracts + state inventory ONLY,
explicitly *not* the prototype, not the conversation — to a fresh-context
subagent. Instruct it to reconstruct the surface (wireframe/session-
transcript fidelity is enough; this is cheap by the same PPR argument that
makes SFD viable). Diff the reconstruction against the converged prototype.

- Divergence = a contract leak → tighten or add a clause; log the leak.
- Repeat once. Budget: max 2 rounds; remaining cosmetic-only diffs are
  accepted and noted in the round-trip report.
- Gate 2 requires the round-trip report. Rule of thumb to print in the
  skill: **contracts are sufficient iff a blind agent can rebuild the
  surface from them.**

### 3.3 Surface State Inventory port + per-cell derivation

Port the whitepaper §7 inventory (10 states per surface unit, classified
in-scope / deferred / not-applicable) into `SKILL.md` Gate 1. Then make
Phase 5 derivation matrix-driven: every in-scope (unit × state) cell maps to
≥1 clause; every clause cites its cells (see `state-cells:` in the Clause
Log entry above). Coverage by construction, auditable by grep.

### 3.4 Checkable at birth, or demoted

A clause without an acceptance procedure does not enter the contracts — it
goes to the open-questions list. Each clause carries an `acceptance:` field
(a test name, a measurable threshold, or a verification command). NFRs need
numbers: "feels fast" → a latency target, or it is explicitly deferred.
This mirrors the downstream ledger's admission gate ("refused: no acceptance
procedure — give me a number, a test, or don't call it a contract") so SFD
output passes that door on the first try regardless of which consumer it
meets.

### 3.5 Propose-Choose-Proceed for contested clauses

Phase 5 currently ends with "Does this capture everything? Anything I'm
missing?" — an open-ended question that violates the skill's own
Anti-Pattern 6. Replace: contested clauses (strict vs loose invariant,
tight vs generous NFR) are presented as 2–3 options with a recommendation,
like every other decision point. Uncontested clauses get batch confirmation.

### 3.6 Negative clauses from rejected alternatives

During Phase 5 compilation, mine the Decision Log: rejected alternatives the
user paid to reject during convergence become "must NOT" clauses where
behaviorally relevant (e.g. "rejected: pie chart for progress → the progress
view MUST NOT use proportional-area encodings"). Provenance points at the
decision entry. Rejections already paid for should never need re-rejecting
at verdict time.

## 5. Acceptance criteria (this work, dogfooded)

1. SKILL.md and whitepaper v0.7 agree — no drift on inventory, gates,
   bundle, or new practices (the skill's own drift rule, satisfied).
2. A fresh-context agent given only the revised SKILL.md and a toy prompt
   produces: a Clause Log, a per-cell derivation, clauses with `acceptance:`
   fields, a round-trip report, and a complete six-artifact Handoff Bundle.
3. Every new mechanic is exercised by at least one eval case in
   `evals/evals.json`, and the dojo baseline shows the old skill failing at
   least one of them (RED before GREEN — proves the change does work).
4. Phases 6–7 are reachable and unchanged behind the demarcation header.
5. No reference to a "contract ledger" as an SFD-internal component —
   ledger-the-sibling stays downstream and out of this repo.
